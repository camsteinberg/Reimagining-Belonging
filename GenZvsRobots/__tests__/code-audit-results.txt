[CRITICAL] app/play/[code]/page.tsx:37-44 - Player ID detection by name match causes wrong-player bug
  The PlayPage detects the current player's ID by matching `p.name === playerName`.
  If two players join with the same name, the first match wins and the second
  player will be bound to the wrong player record. They will see the wrong
  team's grid and role, and their block placements will be attributed to
  someone else.
  Impact: Two players named "Alex" causes one to control the other's session.
          The second player is permanently stuck showing wrong data.
  Fix: Use the PartySocket connection ID (socket.current.id) as the player ID
       instead of matching by name. The server keys players by connection ID,
       so the client should use its own socket ID directly. Example:
         const playerId = socketRef.current?.id;
       This is available from the partysocket library. Pass socket ref from
       usePartySocket and use socket.current.id.

[CRITICAL] party/index.ts:29-35 - Host disconnect marks hostConnected=false but does NOT pause/stop the timer
  When the host disconnects (onClose), `hostConnected` is set to false and
  the player is marked disconnected, but the server timer keeps running.
  If the host disconnects during a round, the round will expire and
  auto-advance to reveal phase with no host to control it. Reconnecting
  host gets a fresh connection ID and can never re-register as host.
  Impact: Host browser crash or network drop during a round = game stuck
          in reveal phase with no way to advance. All players are stranded.
  Fix: Either (a) pause the timer when host disconnects and resume on
       reconnect, or (b) allow a host to re-join by sending a new "join"
       message with isHost=true and mapping to the new connection ID.
       At minimum, store the host identity separately from connection ID
       and allow re-registration.

[CRITICAL] party/gameState.ts:63-67 - removePlayer only marks disconnected, never cleans up team roster
  When a player disconnects, `removePlayer` sets `connected = false` but
  does NOT remove the player ID from `team.players[]`. This means:
  1) Team size is never reduced, so new players cannot fill the slot
  2) If the disconnected player was the only architect, the team has no
     connected architect — nobody can see the target in Round 1
  3) Role swapping in Round 2 still swaps the disconnected player's role
  Impact: Player disconnect permanently reduces effective team size.
          Team with disconnected architect has no one who can see the target.
  Fix: Either remove the player ID from the team.players array and
       reassign architect role to another connected team member, OR
       at minimum, promote the first connected builder to architect
       when the current architect disconnects.

[CRITICAL] party/gameState.ts:86-92 - Role swap in round 2 swaps ALL players including disconnected ones
  The round 2 role swap iterates `Object.values(state.players)` and swaps
  every player's role. If a team had 3 players (1 architect + 2 builders)
  and the architect disconnected, the swap produces 1 disconnected builder
  + 2 architects — but the game expects exactly 1 architect per team.
  With 2 architects on a team, both see the target but neither has the
  builder UI to place blocks.
  Impact: Team with disconnected player gets broken roles in Round 2.
          Could result in 0 builders (nobody can build) or 2 architects.
  Fix: Only swap roles for connected players, and ensure exactly one
       connected player per team is architect after the swap.

[CRITICAL] party/gameState.ts:32-60 - addPlayer creates teams with wrong team count after disconnections
  `addPlayer` calculates team index as `Object.keys(state.teams).length`
  for TEAM_NAMES lookup. Since disconnected players are never removed from
  teams, teams are never "full" once a player disconnects (connected flag
  doesn't affect the `players.length < MAX_TEAM_SIZE` check). But if a
  team somehow fills to MAX_TEAM_SIZE with 3 connected players, new players
  create a new team. This is fine, but the team naming index uses total
  team count, so with 8+ teams you'd wrap around TEAM_NAMES correctly.
  However, a single disconnected player in a team of 2 means the team
  shows 2 players but only 1 is connected. New players will fill a
  different team first (or create a new one) rather than replacing
  the disconnected slot.
  Impact: Unbalanced teams — some teams have ghost players taking up slots.
  Fix: Either skip disconnected players in the team size count, or fully
       remove disconnected players from the team roster.

[MAJOR] party/index.ts:162-169 - AI response applies blocks without checking game phase
  The onRequest HTTP handler applies AI block placements via `placeBlock()`
  but does NOT check that the game is in round2 phase. If the AI API
  response arrives late (after the round ends), blocks could be placed
  during reveal or summary phases. The `placeBlock` function in gameState
  does check phase, but there's a TOCTOU window: if endRound fires between
  the phase check in placeBlock and the actual grid mutation, blocks could
  leak into the wrong phase. More practically, if the round ends and the
  state transitions to "reveal1" just before the AI response arrives,
  placeBlock correctly rejects — but the chat message and aiBuilding
  broadcast on lines 174-190 still fire, showing an AI chat message
  during reveal phase.
  Impact: AI chat messages appear during reveal phase, confusing players.
  Fix: Add a phase check at the top of the aiResponse handler:
       if (state.phase !== "round2") return new Response("Round ended", ...);

[MAJOR] app/api/ai/chat/route.ts:22-27 - AI chat has no conversation history (stateless)
  Each AI request sends only the single latest user message with no
  conversation history. The AI cannot remember previous messages in the
  same round, so it gives repetitive or context-free responses.
  Impact: AI feels robotic and unhelpful — asks players to repeat themselves,
          gives the same descriptions multiple times. Poor UX.
  Fix: Maintain conversation history per team per round, either in-memory
       on the server or by passing the conversation array from the client.

[MAJOR] app/api/ai/chat/route.ts:7-9 - Anthropic client created at module level may crash on missing key
  `new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY })` runs at import
  time. If ANTHROPIC_API_KEY is not set, the Anthropic SDK may throw during
  module initialization, crashing the entire API route (and potentially
  blocking other routes in the same edge function bundle).
  Impact: Missing API key = 500 error on ALL requests to this route,
          with an unhelpful error message.
  Fix: Lazy-initialize the client inside the POST handler, or check for
       the key and return a clear error.

[MAJOR] party/index.ts:246-255 - Server timer resolution is 1 second but checks Date.now() >= timerEnd
  The timer interval fires every 1000ms and checks if `Date.now() >= timerEnd`.
  If the check fires at timerEnd - 500ms, it misses. Next check at
  timerEnd + 500ms, it catches. This means the round can run up to ~1s
  longer than intended. For a 5-minute round this is negligible, but the
  design means the server and client timers can disagree by up to 1 second,
  causing the client to show "00:00" for up to a second before the server
  ends the round.
  Impact: Minor — round runs up to 1s long. Client shows 00:00 briefly
          before server catches up.
  Fix: Reduce interval to 200ms, or use setTimeout with calculated delay
       to fire precisely at timerEnd.

[MAJOR] components/PlayerView.tsx:92-98 - setTimeout cleanup leak on rapid block placement
  `handleCellClick` creates a setTimeout to clear the "new" animation on
  cells. If the component unmounts before the timeout fires (e.g., round
  ends), the timeout still fires and calls setNewCells on an unmounted
  component. React will log a warning in development.
  Impact: React state-update-on-unmounted-component warnings. In rare
          cases, could cause stale state if the same component remounts.
  Fix: Store timeout IDs in a ref and clear them on unmount via useEffect
       cleanup, or use a ref-based approach to skip the setState if
       unmounted.

[MAJOR] lib/usePartySocket.ts:34-41 - JSON.parse on WebSocket message with no error handling
  Line 35: `const msg: ServerMessage = JSON.parse(e.data)` has no try/catch.
  If the server sends malformed JSON (e.g., due to a bug or network
  corruption), this will throw an uncaught exception that crashes the
  message handler. No further messages will be processed.
  Impact: Single malformed message breaks all subsequent message processing
          for that client session. Game appears frozen.
  Fix: Wrap in try/catch and ignore malformed messages.

[MAJOR] components/HostView.tsx:222-226 - Host reveal1 "Next Phase" button sends startRound directly, skipping interstitial
  HostControls for reveal1 phase shows two buttons: "Show Scores" (nextReveal)

  Line 164: `team.round2Score = calculateScore(team.grid, ROUND_2_TARGET)`

[MAJOR] party/index.ts:88-92 - Role swap logic doesn't handle teams with only 1 player
  If a team has exactly 1 player (e.g., the second player disconnected or
  never joined), that player is the architect. On round 2 role swap, they
  become a builder. Now the team has 0 architects and 1 builder. The builder
  can place blocks but nobody can see the target to guide them.
  Impact: Solo-player team in round 2 has no architect — they're blind.
  Fix: If a team has only 1 connected player, keep them as architect
       (or assign them both roles / show them the target regardless).

[MAJOR] components/RoundTransition.tsx:170 - window?.innerHeight used in animation causes SSR mismatch
  Line 170: `y: [0, -(window?.innerHeight ?? 900) - 40]` reads
  window.innerHeight during render. This value is baked into the animation
  at the time the component mounts. If the window resizes, particles will
  float to the wrong height. More importantly, during SSR/hydration,
  window is undefined, so the fallback 900 is used, potentially causing
  a hydration mismatch if the client window height differs.
  Impact: Particle animation height may not match viewport. Minor visual
          issue but could cause React hydration warning.
  Fix: Use a ref to track window height and update on resize, or use
       CSS viewport units (100vh) instead of a JS value.

[MAJOR] party/index.ts:248-254 - Timer expiry and host "skipToReveal" can race
  If the host clicks "Skip to Reveal" at the exact moment the timer expires,
  both `handleHostAction("skipToReveal")` and the timer callback will call
  `calculateAllScores` and `endRound`. Since JavaScript is single-threaded
  in Node.js, these won't actually interleave, but the timer callback
  could fire after `endRound` already transitioned the phase. The second
  call to endRound would be a no-op (phase is no longer round1/round2),
  and calculateAllScores would run twice (harmless but wasteful).
  Impact: Harmless double-calculation. Scores are idempotent.
  Fix: Check phase in the timer callback before calling endRound.

  Fix: No fix needed unless team reassignment is added later.

[MINOR] party/index.ts:40 - Client messages parsed with unchecked `as ClientMessage` cast
  Line 40: `JSON.parse(message) as ClientMessage` — no runtime validation
  of the message shape. A malicious client could send arbitrary JSON that
  doesn't match ClientMessage, and the switch statement would hit the
  default case (no-op). Most fields are checked before use (e.g., player
  lookup), so this is low risk.
  Impact: Low — malformed messages are ignored or cause no-ops.
  Fix: Acceptable for a game, but adding zod/valibot validation would
       make it more robust.

[MINOR] party/index.ts:163-170 - AI response HTTP endpoint has no authentication
  The onRequest handler accepts POST requests from anywhere (CORS: *)
  with no authentication. Anyone who knows the room code could send
  fake AI responses to place blocks and inject chat messages.
  Impact: Cheating / griefing vector — attacker could place arbitrary
          blocks or send fake "Scout" messages.
  Fix: Add a shared secret or signature between the Next.js API route
       and the PartyKit server. For a game jam project this is low priority.

[MINOR] app/api/ai/chat/route.ts:13 - No rate limiting on AI chat endpoint
  The AI chat endpoint calls the Anthropic API with no rate limiting.
  A malicious client could spam the endpoint, running up API costs.
  Impact: Potential API cost abuse.
  Fix: Add rate limiting per team/player, or throttle on the client side.
